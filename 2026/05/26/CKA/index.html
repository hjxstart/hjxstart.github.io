Linux系统架构

## 内核(kernel)
Linux内核作为操作系统的核心组件，直接与硬件交互、负载管理CPU、内存、设备驱动等底层资源。它通过高效的进程调度算法实现多任务并行，并提供完善的内存管理和系统调用接口。
## Shell解释器
Shell作为命令行解释器，不仅提供用户与系统交互的接口，还支持脚本编程。它能解析用户输入的命令，调用相应的系统功能，并提供管道、重定向等高级特性，大大提升了系统使用效率。
## 用户空间(User Space)
用户空间是应用程序运行的环境，包含系统服务、开发工具和应用软件。在这里，程序运行在受包含的内存空间中，通过系统调用与内核通信，确保了系统的安全性和稳定性。

1.2 容器技术入门

容器技术概述

# 定义: 容器是一种轻量级、可移植的软件打包技术，将应用及其依赖封装为独立单元。
# 核心技术: Linux内核特性: 命名空间(Namespaces)、控制组(cgroups)、Union文件系统。
# 主要优势: 轻量高效、一致环境、快速部署、弹性伸缩、高密度运行。
# 应用场景: 微服务架构、持续集成/持续部署、云原生应用、开发与测试环境。

虚拟化技术概述

# 硬件虚拟化: 直接在硬件上模拟完整系统
# 操作系统虚拟化: 共享主机内核，隔离用户空间
# 应用虚拟化: 封装应用程序及其依赖

容器vs虚拟机

## 架构对比
# 虚拟机架构: 物理硬件->宿主机OS->Hypervisor->多个完整OS->应用
1. 每个VM包含完整操作系统
2. 通过Hypervisor管理
3. 资源消耗较大
4. 启动时间长(分钟级)
# 容器架构: 物理硬件->宿主机OS->容器引擎->多个应用及依赖
1. 共享宿主操作系统内核
2. 每个容器只包含应用和依赖
3. 轻量化设计
4. 启动时间短(秒级)

## 资源利用率对比，容器都比较低
内存
磁盘
启动时间
CPU开销

## 安全性对比
# 虚拟机安全特性
完全隔离的操作系统，独立内核，硬件级别分离，更强的边界防护。
# 容器安全特性
共享内核带来风险，进程级隔离，依赖内核安全特性，需额外安全措施。
# 安全权衡
虚拟机提供更高安全性但效率较低，容器轻量高效但隔离较弱。


## 可移植性对比
# 打包方式
容器：轻量级镜像，MB级别，包含用户和依赖
虚拟机：完整磁盘镜像，GB级别，包含整个操作系统
# 环境依赖
容器：需要兼容的宿主机内核，存在一定限制
虚拟机：完全独立，对宿主机系统几乎依赖
# 迁移难度
容器：传输快速，配置简单，秒级部署
虚拟机：传输慢，配置复杂，分钟级部署


## 容器技术的优势
1. 轻量高效：极小的资源占用，快速启动和部署
2. 一致环境：“在我的机器上能运行”问题的解决方案
3. 分层文件系统实现高效存储和复用
4. 弹性伸缩：快速水平扩展，按需分配资源
5. 开发友好：简化开发、测试和部署流程


## 容器技术的挑战
安全隔离：容器共享内核，隔离度不如虚拟机，存在潜在安全风险。
监控与管理：大规模容器集群的监控、日志收集和故障排查较复杂。
持久化存储：容器本身无状态，持久化数据需要额外解决方案。

2.1 docker

Docker的安装

# 选择
Docker支持Windows10 专业版或企业版、macOS 10.15或更高版本，以及主流Linux发行版。请确保您的系统满足最低要求：4GB内存，至少10GB可用磁盘空间，并启用虚拟化功能。
# 下载安装
访问docker.com/products/docker-desktop，选择您的操作系统对应的"Downlaod"按钮。对于Windows和MAC用户，推荐下载Docker Desktop稳定版；Linux用户可以使用包管理器或下载DEB/RPM包进行安装。

运行安装程序

# 安装步骤
双击下载的Docker安装程序(.exe或.dmg文件)启动安装向导。在安装过程中，您需要:
1. 仔细阅读并接受许可协助；
2. 选择合适的安装位置（建议使用默认路径）
3. 根据系统提示选择所需组件，如Docker Desktop、Docker Compose等；
4. 根据需要配置系统环境变量。
# 完成安装
安装完成后，系统会自动启动Docker服务。您可以：
1. 打开Docker Desktop应用程序检查运行状态；
2. 在终端中输入"docker --version"验证安装；
3. 运行"docker run hello-world"测试Docker功能是否正常。
如遇到问题，请查看Docker日志或访问官方文档获取帮助。
# 打开终端
在Windows系统中打开PowerShell或命令提示符，在Mac/Linux系统中打开终端。输入命令"docker version"或"docker --version"来检查安装状态。
# 查看版本信息
如果安装成功，系统将显示Docker客户端和服务器的详细版本信息，包括版本号、构建时间等。如果出现命令未找到的错误，请确保Docker服务已启动并重新安装。

Docker基础操作

# 1. 镜像
一个只读的模板文件，它是容器的基础。包含完整的运行环境，包括应用程序、系统工具、系统库和配置文件。
# 2. 容器
镜像的可运行实例，提供独立的运行空间和资源隔离。可以启动、停止、删除和暂停容器，实现应用程序的快速部署。
# 3. Docker Hub
官方的镜像仓库平台，存储和分享Docker镜像。用户可以上传自己的镜像或使用其他开发者共享的镜像，加速开发流程。

拉取Docker镜像

# Docker pull 命令
使用"docker pull [镜像名称]:[标签]"命令从Docker Hub拉取镜像。标签是可选的，如果不指定，默认拉取latest版本。拉取操作会自动下载并验证镜像的完整性，确保安装可靠。
# 示例
例如，要拉取特定版本的Nginx镜像，可以使用"docker pull nginx:1.24.0"命令。如果只需要最新版本，直接使用"docker pull nginx"即可。拉取完成后，可以使用"docker images"命令查看本地镜像列表。

运行Docker容器

# Docker run命令
使用"docker run [镜像名称]"命令可以创建并启动一个新的容器。这是最基本的Docker操作之一，可以通过添加不同的参数来自定义容器的运行方式。
# 示例
例如，要运行Nginx镜像，可以使用"docker run -d -p 80:80 nginx"命令。其中"-d"表示在后台运行容器，"-p 80:80"表示将容器的80端口映射到主机的80端口，"nginx"是镜像名称

容器端口映射

# 容器端口
容器端口的应用程序会在特定端口上运行服务，比如Web服务器通常使用80端口，数据库可能使用3306端口。这些端口在容器内部是独立的环境。
# 主机端口映射
为了从外部访问容器服务，需要使用"-p"参数将容器端口映射到主机端口。例如，"-p 8080:80"表示容器的80端口映射到主机的8080端口，这样就可以通过访问主机的8080端口来访问容器服务。

查看正在运行的容器

# Docker ps命令
使用"docker ps"命令可以实时查看系统中所有正在运行的容器列表。添加"-a"参数可以同时显示已停止的容器，这对于监控和管理容器状态非常有帮助。
# 输出信息
命令执行后会显示详细的容器信息，包括：容器ID、使用的镜像名称、创建时间、当期状态、暴露的端口映射，以及容器的名称。这些信息对于容器管理和故障排查至关重要。

停止Docker容器

# Docker stop 命令
使用"docker stop [容器ID]"命令可以优雅地停止正在运行的容器。这个命令会先向容器内的主进程发送SIGTERM信号，允许应用程序完成正在进行的任务并进行清理。如果容器在10秒内没有停止，Docker将发送SIGKILL信号强制终止容器。
# 示例
例如，要停止容器ID为"c234567890"的容器，可以使用"docker stop c234567890"命令。您也可以使用容器名称来代替容器ID，如"docker stop myapp"。如果需要同时停止多个容器，可以在命令中列出多个容器ID或名称。

启动Docker容器

# Docker start命令
使用"docker start [容器ID]"命令可以启动已停止的Docker容器。此命令支持同时启动多个容器，只需在命令后列出多个容器ID即可。启动后，容器将恢复到停止前的状态，保留所有配置和数据。
# 示例
例如，要启动容器ID为"c234567890"的容器，可以使用"docker start c234567890"命令。启动后可以使用"docker ps"命令验证容器状态，确保容器已成功运行。如果要同时启动多个容器，可以使用"docker start c234567890 d123456789"这样的格式。

删除Docker容器

# Docker rm命令
使用"docker rm [容器ID]"命令可以永久删除指定的Docker容器。删除前请确保容器已经停止运行。可以添加"-f"参数强制删除正在运行的容器，但这可能会导致数据丢失。
# 使用示例
1. 删除已停止的容器："docker rm c234567890"
2. 强制删除运行中的容器："docker rm -f c234567890"
3. 删除所有已停止的容器: "docker rm $(docker ps -aq)"

查看Docker镜像

# Docker images命令
使用"docker images"命令可以列出本地系统中所有已下载和构建的Docker镜像。此命令简单直观，是管理Docker镜像的基础命令之一。
# 输出信息
命令执行后会显示详细的镜像信息，包括: PEPOSITORY （镜像的仓库源），TAG（镜像的标签版本），IMAGE ID（镜像的唯一标识符），GREATED（镜像的创建时间）以及SIZE（镜像的大小）。这些信息有助于识别和管理系统中的Docker镜像。

删除Docker镜像

# Docker rm 命令
使用"docker rmi [镜像ID]"命令可以删除不在需要的Docker镜像，从而释放系统存储空间。删除前请确保没有容器正在使用该镜像，否则删除镜像将失败。
# 使用示例
删除指定镜像：使用"docker rm d23456780"命令。您也可以使用镜像名称和标签，如"docker rmi nginx:latest"。如果要强制删除，可以添加-f参数。

Dockerfile简介

# 定义镜像构建过程
Dockerfile是一个用于自动化构建Docker镜像的配置文本文件。它包含一系列清晰的指令和参数，按照从上到下的顺序执行，每条指令都会在镜像中创建一个新的层。通过这种方式，可以精确控制镜像的构建过程。
# 构建自定义镜像
通过编写Dockerfile，开发者可以创建完全定制化Docker镜像。这包括选择基础操作系统、安装必要的依赖包、配置运行环境、复制应用程序代码等。这种方式确保了镜像构建过程的可重复性和一致性，非常适合持续集成和部署。

Dockerfile指令

# 1. FROM
指定构建镜像的基础层，是每个Dockerfile的第一条指令。例如："FROM ubuntu:20.04" 或 "FROM node:14-alpine"。
# 2. RUN
在镜像中执行shell命令来安装依赖或配置环境。例如："RUN apt-get update && apt-get install -y python3"。支持多行命令。
# 3. COPY
从构建上下文复制文件或目录到镜像中。常用于部署应用代码，例如: "COPY ./src /user/src/app"。比ADD指令更透明和简单。
# 4. CMD
定义容器启动时的默认命令。每个Dockerfile只能有一个生效的CMD指令。例如: "CMD ["npm", "start"]" 或 "CMD ["python3", "app.py"]"

构建Docker镜像

# Docker build命令
使用"docker build"命令从Dockerfile构建镜像。常用参数包括: -t 设置镜像标签，--no-cache不适用缓存，-f指定Docker路径。构建过程会逐步执行Dockerfile中的每条指令。
# 示例
基本构建命令: "docker build -t myapp:1.0"。生产环境推荐使用版本标签，如: "docker build -t myapp:latest -t myapp:1.0.1"。添加--no--cache参数可以确保获取最新依赖: "docker build --no--cache -t myapp"。

Docker Compose是一个强大的工具，用于简化多容器Docker应用程序的开发和部署。它让您能够使用单个命令同时管理多个相互关联的容器，大大提高了开发效率和部署可靠性。

Docker网络简介

# 容器间通信
Docker网络提供了灵活的容器通信机制，确保容器能够安全、高效地进行数据交换。它支持容器间的DNS解析，并允许容器通过访问发现相互访问。
# 网络类型
Docker支持多种网络驱动类型：bridge用于单主机容器通信，host直接使用主机网络栈，overlay支持跨主机容器通信，macvlan允许容器直接连接物理网络。

桥接网络

# 默认网络
桥接网络是Docker的默认网络类型，它创建一个虚拟网桥，使同一网络中的容器能够自动发现并相互通信。每个容器都会获得唯一的IP地址，便于网络管理和服务发现。
# 隔离网络
桥接网络通过网络命令空间实现容器间的网络隔离，为每个容器提供独立的网络栈。这种隔离机制不仅确保了容器通信的安全性，还支持自定义网络策略和端口映射，使容器与外部网络的交互更加灵活可控。

主机网络

# 共享主机网络
主机网络模式让容器直接使用主机的网络栈，容器可以无限制地使用主机的所有网络端口。这种模式下，容器的网络性能最好，因为不需要网络地址转换（NAT）和额外的路由开销。特别适用于对网络性能要求极高的应用场景。
# 无网络隔离
在主机网络模式下，容器于主机共享网络命名空间，可以直接访问主机的网络接口和服务。虽然这提供了最大的灵活性，但需要主要安全风险，建议仅在受信任的环境中使用。对于需要严格网络控制的生产环境，推荐使用其他网络模式。

overlay网络

# 跨主机通信
overlay网络是一种高级网络驱动，能够让不同Docker主机上的容器进行透明通信。它通过VXLAN技术在物理网络之上构建虚拟网络层，确保容器通信安全且高效。
# 扩展性和高可用性
overlay网络支持大规模容器集群部署，可动态添加或移除节点。通过内置的服务发现和负载均衡机制，它能自动处理容器故障转移，提供接近零停机时间的服务可用性。

Docker数据卷简介

# 持久化数据
Docker数据卷（Volume）是独立于容器的特殊文件系统，用于实现数据持久化存储。它完全由Docker管理，不受容器生命周期的影响，确保在容器删除后数据依然安全保持。
# 共享数据
数据卷支持多容器同时挂载和访问，可实现容器间的数据共享和实时同步。这种机制特别适用于开发团队协作、数据备份和微服务架构中的数据交互场景。

数据卷创建和挂载

# 创建数据卷
使用"docker volume create [卷名称]"命令创建新的数据卷。这是管理Docker持久化存储的第一步，创建后的数据卷可以被多个容器反复使用。系统会自动在默认位置生成对应的存储空间。
# 挂载数据卷
使用"-v [卷名称]:[容器路径]"选项将数据卷挂载到容器的特定路径。这个操作建立了容器于数据卷之间的连接，使容器可以读写数据卷中的内容。挂载后的数据会实时同步，便于管理和备份。

数据卷使用场景

# 数据库数据
将数据卷用于存储MySQL、MongoDB等数据库的数据文件，可以确保容器重启或更新时数据不会丢失，同时便于多个容器之间共享数据。特别适合开发和生成环境的数据持久化需求。
# 配置文件

## Docker对内存的使用限制
1. 与操作系统类似，容器可以使用的内存包括两部分：物理内存和swap。
2. Docker通过下面两组参数来控制容器内存的使用量
	m 或者 --memory: 设置内存的使用限额，例如100M, 2G。
	-memory-swap: 设置内存+swap的使用额度。
3. 示例: docker run -m 200M --memory-swap=300M ubuntu # 含义是允许容器最多使用200M的内存和300M的swap
4. 使用progrium/stress镜像来学习如何为容器分配内存。该镜像可用于对容器执行压力测试。执行如下命令:
docker run -it -m 200M --memory-swap=300M nginx --vm 1 --vm-bytes 280M
	-vm 1: 启动一个内存工作线程
	-vm-bytes 280M: 每个线程分配280M内存
5. 因为280M 在可分配的范围(300M)内，所有工作线程能够正常工作，其过程是：
分配280M内存
释放280M内存
再分配280M内存
再释放280M内存
一直循环
6. 如果让工作线程分配的内存超过300M
docker run it -m 200M --memory-swap=300M progrium/stress --vm 1 --vm-bytes 310M
	-vm 1: 启动1个内存工作线程
	-vm-bytes 310M: 每个线程分配310内存
7. 分配的内存超过限额，stress线程报错，容器退出。
如果在启动容器时只指定-m而不指定 --memory-swap，那么--memory-swap默认为-m的两倍，比如
docker run -it -m 200M ubuntu 
容器最多使用200M物理内存和400M swap

## Docker对CPU的使用限制
默认设置下，所有容器可以平等地使用 host CPU 资源并且没有限制。
Docker可以通过-c或者--cpu-shares设置容器使用cpu的权重。如果不指定，默认值为1024.
与内存限额不同，通过-c设置的cpu share并不是CPU资源的绝对数量，而是一个相对的权重值。某个容器最终能分配到的CPU资源取决于它的cpu share占所有容器的cpu share总和的比例。换句话说：通过cpu share可以设置容器使用CPU的优先级。
1. 比如在host中启动了两个容器：
docker run --name "container_A" -c 1024 ubuntu
docker run --name "container_B" -c 512 ubuntu
container_A的CPU share 1024，是container_B的两倍。当两个容器都需要CPU资源时，container_A可以得到的CPU是container_B的两倍。
2. 需要特别注意的是：这种按权重分配CPU只会发生在CPU资源紧张的情况下。如果container_A处于空闲状态，这时，为了充分利用CPU资源，container_B也可以分配到全部可用的CPU。
# 继续使用progrium/stress做实验
1. 启动container_A, cpu share 为1024
docker run --name container_A -it -c 1024 progrium/stress --cpu 1
	-cpu用来设置工作线程的数量。因为当前host只有1课CPU，所以一个工作线程就能将CPU压满。如果host有多颗CPU，则需要相应增加--cpu的数量。
2. 启动container_B, cpu share为512：
docker run --name container_B -it -c 512 progrium/stress --cpus1
3. 在host中执行top，查看容器对CPU的使用情况：container_A消耗的CPU是container_B的两倍
4. 现在暂停container_A:
docker pause container_A
5. top显示container_B在container_A空闲的情况下能够用满整颗CPU:


## Docker对I/O的使用
1. Block IO是另一种可以限制容器使用的资源。Block IO指的是磁盘的读写，docker可通过设置权重、限制bps和iops的方式控制容器读写磁盘的带宽，下面分别讨论。
2. 默认情况下，所有容器能平等地读写磁盘，可以通过设置--blkio-weight参数来改变容器block IO的优先级。
	-blkio-weight与--cpu-shares类似，设置的是相对权重值，默认为500。在下面的例子中，container_A读写磁盘的带宽是container_B的两倍。
	docker run -it --name container_A --blkio-weight 600 ubuntu
	docker run -it --name container_B --blkio-weight 300 ubuntu
3. 限制bps和iops
	bps是byte per second, 每秒读写的数据量。
	iops是io per second, 每秒IO的次数。
	可通过以下参数控制容器的bps和iops:
		-device-read-bps, 限制读某个设备的bps
		-device-write-bps, 限制写某个设备的bps
		-device-read-iops, 限制读某个设备的iops
		-device-write-iops, 限制写某个设备的iops
		下面这个例子限制容器写/dev/sda的速率为30MB/s
		docker run -it --device-write-bps /dev/sda:30MB centos
	通过dd测试在容器中写磁盘的速度。因为容器的文件系统在 host /dev/sda 上的，在容器中写文件相当于对 host /dev/sda进行写操作。另外，oflag=direct指定用direct IO方式写文件，这样--device-write-bps才能生效。
	如果不限速，测试命令如下
		docker run -it centos
		dd if=/dev/zero of=test.out bs=1M count=1000 oflag=direct

容器访问控制

容器的访问控制主要通过Linux上的iptabes防火墙软件进行管理和实现。
## 容器访问外部网络
1. 容器默认指定了网关docker0网桥上的docker0内部接口。docker0内部接口同时也是宿主机的一个本地接口。因此，容器默认情况下可以访问到宿主机本地网络。如果容器要想通过宿主机访问到外部网络，则需要宿主机进行辅助转发。
2. 在宿主机Linux系统中，检查转发是否开启，代码如下：
	sysctl net.ipv4.ip_forward=1
	如果为0，说明没有开启转发，则需要手动打开：
	sysctl -w net.ipv4.ip_forward=1
	Docker服务启动时会默认开启--ip--forward=true，自动配置宿主机系统的转发规则。

## 容器之间访问
容器的访问控制主要通过Linux上的iptables防火墙进行管理和实现
# 容器之间相互访问需要两方面的支持
1. 网络拓扑是否已经连通。默认情况下，所有容器都会连接到docker0网桥上，这意味着默认情况下拓扑是互通的
2. 本地系统的防火墙软件iptabes是否允许访问通过。这取决于防火墙的默认规则是允许（大部分情况）还是禁止。
# 下面分两种情况介绍容器间的访问
1. 访问所有端口
	1）当启动Docker服务时候，默认会添加一条“允许”转发策略到iptabes的forWARD链上。通过配置--icc=true|false（默认值为true）参数可以控制默认的策略。
	2）为了安全考虑，可以在Docker配置文件中配置DOCKER_OPTS=--icc=false来默认禁止容器之间的互相访问。
	3）同时，如果启动Docker服务时手动指定--iptables=false参数，则不会修改宿主机系统上的iptables规则。
2. 访问指定端口
	1）在通过--icc=false禁止容器间相互访问后，仍可以通过--link=CONTAINER_NAME:ALLAS选项来允许访问指定容器的开发端口。
	2）例如，在启动Docker服务时，可以同时使用icc=false --iptables=true参数来配置容器间禁止访问，并允许Docker自动修改系统中的iptables规则。此时，系统中的iptables规则可能是类似如下规则，禁止所有转发流量：

## 数据卷

1. docker 为容器提供了两种存储资源：数据层和Data Volume
2. 数据层包括镜像层和容器层，由storage driver管理
3. Data Volume有两种类型：bind mount 和 docker managed volume
4. bind mount 可以实现容器与host之间，容器与容器之间共享数据
5. volume container是一种具有更多移植性的容器间数据共享方案，特别是data-packed volume container
6. 容器中的数据管理方式
	1）数据卷：容器内数据直接映射到本地主机环境
	2）数据卷容器：使用特定的容器维护数据

## 数据卷实操
1. 数据卷是一个可供一个或多个容器使用的特殊目录，他将主机操作系统目录直接映射到容器，类似 mount
2. 数据卷可以在容器之间共享和重用
3. 对数据卷的修改会立马生效
4. 对数据卷的更新，不会影响镜像
5. 数据卷默认会一直存储，即使容器被删除
## 实操
1. 创建数据卷
docker volume create -d local test #  数据卷路径 /var/lib/docker/volumes/
2. 绑定创建的数据卷 (在创建容器时将主机本地的路径挂载到容器内作为数据卷)
docker run命令时，使用-mount选项使用数据卷
# 挂载刚刚创建的数据卷到容器中--mount source=要挂载的数据卷,target=挂载到容器的路径 镜像名
docker run -d -P --name nginx --mount source=test,target=/webapp nginx
# 默认挂载的镜像是读写权限，可以设置为只读
docker run -d -P --name nginx --mount source=test,target=/webapp:ro nginx
# 直接挂载主机目录到容器（业务测试场景）
docker run -it -v /宿主机绝对目录:/容器内目录 镜像名


## 数据卷容器
本身就是一个容器，专门提供数据卷给其他容器，用于多个容器数据共享。
1. 创建数据卷容器，并将一个数据卷挂载到此容器下
docker run -it -v /dbdata --name db1 ubuntu
2. 创建db1和db2都挂载到同一个数据卷/dbdata目录下
docker run -it --volumes-from dbdata --name db1 ubuntu
docker run -it --volumes-from dbdata --name db2 ubuntu
向三个容器中任意一个/dbdata目录下写入，其他容器都可以看到
3. 可以使用--volumes-from参数来从多个容器挂载多个数据卷，也可以从其他以挂载的数据卷容器来挂载数据卷
docker run -d --name db3 --volumes-from db1 training/postgres
在/dbdata目录下也可以看到文件
4. 如果删除了挂载卷的容器，数据卷不会自动删除，如果要删除数据卷，必须在删除最后一个挂载着他的容器时使用docker rm -v 命令指定数据卷。

## 利用数据卷容器来迁移数据

1. 备份
docker run --volumes-from dbdata -v $(pwd):/backup --name worker ubuntu tar cvf /backup/backup.tar /dbdata
	--name worker ubuntu # 使用ubuntu镜像创建名为worker的容器
	--volumes-from dbdata # 让worker容器挂载dbdata容器的数据卷
	-v $(pwd):/backup # 挂载本地的当前目录到worker容器的/backup目录
	tar cvf /backup/backup.tar /dbdata # 将/dbdata下内容备份为容器的/backup/backup.tar
2. 恢复
	1) 创建一个带有数据卷的容器dbdata2
	docker run -v /dbdata --name dbdata2 ubuntu /bin/bash
	2) 创建另一个新的容器，挂载dbdata2的容器，并解压备份文件到所挂载容器中
	docker run --volumes-from dbdata2 -v $(pwd):/backup busybox tar -xcf /backup/backup.tar

Kubernetes(通常简称为K8s)是当今最受欢迎的容器编排平台，它彻底改变了现代应用程序的部署方式。作为一个功能强大的开源系统，它不仅可以自动化容器化应用程序的部署和扩展，还能智能地管理工作负载。凭借其强大的自动化能力，开发团队可以专注于应用程序开发，而将复杂的容器管理、负载均衡、服务发现和故障恢复等基础设施问题交给Kubernetes处理，从而显著提供开发效率和系统可靠性。

# 容器(Container)
容器是一个标准化的软件单元，将应用程序代码和所有依赖项（包括运行时、系统工具和库）打包在一起。它通过实现完整的运行时环境隔离，确保应用程序可以在任何环境中一致地运行。每个容器都在独立的用户空间运行，共享主机操作系统的内核。

# Pod
Pod是Kubernetes中最基础的调度和资源单位，它代表了集群中的一个运行实例。一个Pod中的多个容器共享相同的网络命名空间、IP地址和存储卷，它们之间可以通过localhost进行通信。Pod设计体现了“边车”(sidecar)模式的概念，使主应用容器能够与辅助容器紧密协作。

# Master节点
作为集群的大脑，负责整体控制和决策。主要功能包括API服务、调度系统、控制器管理以及集群状态存储。它确保整个集群按照预期配置进行，并处理系统级别的自动化任务。

# Node节点
作为集群的工作单元，负责实际工作负载的运行。每个Node都运行着容器运行时（如Docker）、网络代理和控制代理，确保应用程序能够正确并与其他服务进行通信。

Kubernet主要组件

# Kubernetes API Server
集群的核心控制中枢，处理所有API请求，负责组件间的通信和认证授权。
# etcd
高可用的分布式键值数据库，持久化存储集群的所有配置数据和状态信息。
# kube-scheduler
智能调度器，根据资源需求、硬件限制、策略约束等条件，将Pod分配到最合适的节点。
# kubelet
节点代理程序，确保容器在Pod中健康运行，负责维护容器的生命周期管理。
# kube-proxy
网络代理组件，维护节点的网络规则和负载均衡，确保Pod网络通信的顺畅性。

Pod的生命周期

# 1. 创建
Pod被调度器创建并分配资源，kube-scheduler根据资源需求选择最合适的Node。
# 2. 启动
kubelet接受到指令，开启拉取容器镜像并按顺序初始化Pod内的容器。
# 3. 运行
Pod进入稳定运行状态，kubelet持续监控容器健康状态并进行管理。
# 4. 成功
所有容器均正常运行，Pod的健康检查通过，成功提供预期的服务。
# 5. 失败
容器运行出现异常，如内存溢出、健康检查失败或崩溃，触发重启策略。
# 6. 终止
接收到删除命令后，Pod优雅终止：先停止接受新请求，完成现有请求后再关闭容器。

Kubernetes命名空间（Namespace）

# 作用
命名空间是Kubernetes中实现多租户的核心机制，可以将集群资源划分为相互隔离的虚拟组。它支持团队之间的协作，让不同项目组可以在同一个集群中独立工作，互不干扰。通过命名空间，管理员可以为不同团队分配资源配额，实现细粒度的访问控制。
# 优势
命名空间代理多重优势：首先，通过资源隔离增强了安全性，避免意外操作影响其他环境；其次，支持精确的资源限制和配额管理，有效预防资源争用；第三，简化了多环境管理，可轻松区分开发、测试和生产环境；最后，支持细粒度的权限控制，确保团队智能访问自己的资源。

Deployment资源

# 概念
Deployment是Kubernetes中最常用的工作负载资源，它不仅用于定义和管理Pod的生命周期，还能确保应用程序始终按照期望的方式运行。通过Deployment，您可以精确控制多个Pod副本的创建、销毁和更新过程。
# 功能
支持声明式的配置管理，可实现多种高级部署策略：零停机的应用更新、自动的弹性伸缩、版本回滚，以及健康状态检查。
Deployment还能确保在系统更新或节点故障时自动维护指定数量的Pod副本。

Service资源

# 用途
Service作为关键的网络抽象层，可以为一组Pod提供统一的访问入口。它能够自动处理Pod的动态变化，确保应用程序的稳定访问，同时实现请求的自动负载均衡分发。
# 类型
Kubernetes支持多种灵活的Service类型：ClusterIP用于集群内部通信，NodePort允许从集群外访问，LoadBalancer提供云平台负载均衡集成，而Ingress则支持更复杂的HTTP路由规则。每种类型都针对不同的使用场景进行了优化。

Ingress资源

# 功能
提供一种HTTP/HTTPS级别的智能路由机制，通过URL路径、主机名等规则，将外部流量精确导向集群的不同Service。支持SSL/TLS终止，请求重写和URL重定向等高级功能。
# 优势
大幅降低了管理和维护成本，无需为每个服务配置独立的负载均衡器。通过集中的访问控制和认证机制增强安全性，同时支持自动化的证书管理和流量监控。

Kubernetes工作负载

# Deployment
管理无状态应用程序的部署和扩展。自动处理Pod的创建、更新和恢复，例如运行Web服务器或API后端，确保服务持续可用。
# StatefulSet
管理有状态应用程序，为每个Pod提供持久化存储和唯一网络标识。适用于数据库、消息队列等需要稳定存储和网络身份的应用。
# DaemonSet
在集群的每个节点上运行特定的Pod副本。常用于部署监控代理、日志收集器或网络插件等基础设施组件。
# Job
执行一次性批处理任务，如数据迁移、文件处理或备份操作。任务完成后自动清理资源，支持并行处理和失败重试。
# CronJob
按照预定时间表自动运行Job。适用于数据库备份、报告生成、邮件发送等周期性任务，支持标准的Cron表达式配置。

Kubernetes存储

# PersistenVolume (PV)
集群级别的存储资源抽象，支持多种底层存储系统如NFS、AWS EBS或Ceph。独立于Pod生命周期，确保数据持久化存储。
# PersistentVolumeClaim (PVC)
用户对存储的申请声明，自动匹配合适的PV资源。可指定容量大小和访问模式，实现存储资源的动态分配。
# StorageClass
提供动态存储配置的模板，管理员可定义不同性能等级（如SSD、HDD）和供应商特定参数。支持按需自动创建PV简化存储管理。

Kubernetes安全

# 角色和权限
使用RBAC (Role-Based Access Control)实现精细化的访问控制。通过定义Role(角色)和RoleBinding(角色绑定)，可用严格限制用户或服务账号对集群资源的操作权限，确保安全隔离。支持命名空间级别和集群级别的权限管理，有效防止未授权访问。
# 网络策略
使用NetworkPolicy创建细粒度的网络访问控制规则。可用基于命名空间、标签选择器和端口定义入站/出站流量策略，实现Pod之间的网络隔离。这种机制有效防止未经授权的网络通信，提供多层次的安全防护，特别适用于多租户环境。

Kubernetes日志管理

4.2 kubernetes概述与架构

微服务架构：基础于设计原则

# 微服务架构概述
微服务架构是一种革新性的软件设计方法，它将大型企业应用优雅地拆分为独立运行的服务单元。每个服务都像积木一样，既能独立存在，又能完美组合。
在这种架构下，每个服务器都专注于解决特定的业务问题，拥有专属的数据存储和处理逻辑。这种精细化的划分不仅提升了代码的可维护性，还显著提高了团队的开发效率。
# 微服务设计原则
单一职责原则：每个服务专注于一个核心业务功能，就像精密机器中的每个齿轮，都有其独特而明确的作用。
服务自治原则：通过标准化的REST API和消息队列实现服务间通信，确保服务边界清晰，避免互相干扰。
弹性设计原则：采用先进的容错机制和熔断器模式，确保系统的稳定性，即使单个服务出现问题，也不会影响整体运行。
灵活扩展原则：根据业务需求，能够精确地对特定服务进行扩展，实现资源的高效利用。

微服务架构的优势和挑战

Kubernetes架构与核心概念

# Master节点
作为集群的大脑和控制中枢，全面管理集群运行状态。核心组件包括处理API请求的API Server、执行调度决策的Scheduler、维护期望状态的Controller Manager，以及存储集群配置和状态信息的分布式键值存储系统etcd。

# Worker节点
作为集群的计算资源提供者，负责实际工作负载的运行。每个节点都运行着容器运行时引擎、管理容器生命周期的kubelet代理，以及实现服务网络功能的kube-proxy组件，共同确保应用程序的稳定运行。

# Pod
Kubernetes中最小的可部署计算单元，代表着一个应用实例。Pod内的多个容器共享网络空间、存储资源和进程命名空间，它们紧密协作，形成一个完整的应用服务单元，便于统一管理和扩展。

# Service
提供可靠的服务发现和负载均衡机制的抽象层。通过标签选择器智能匹配目标Pod，为应用提供稳定的网络端点，在Pod动态伸缩或故障时自动维护服务的连续性，确保微服务架构的可靠通信。

Kubernetes中的Pod与Service

# Pod
Pod是Kubernetes中最基础的调度和部署单元，可用视为一个轻量级的虚拟机。
每个Pod都拥有独立的集群内IP地址，使得Pod内的容器能够共享网络空间并实现高效通信。
Pod可用封装一个或多个紧密耦合的容器，这些容器共享存储卷、网络命令空间和计算资源配额。

自动实现请求的负载均衡分发，确保流量均匀分布到所有监控的后端Pod。

Kubernetes中的Deployment与StatefulSet

# Deployment
管理无状态应用的理想选择，提供零停机更新和便捷回滚功能；自动维护指定数量的Pod副本，智能处理Pod的调度与自愈；完美支持Web应用、API服务等无状态工作负载；Pod具有动态特性：名称随机生成，每次重启可能分配新的IP地址。

# StatefulSet
专为有状态应用设计，确保Pod的稳定性与启动顺序；每个Pod都拥有唯一且持久的网络标识和专属存储；最适合部署数据库、消息队列等需要数据持久化的应用；Pod具有稳定特性：名称固定且有序，重启后保持相同的网络身份。

为Redis、MongoDB等有状态应用提供稳定可靠的部署方案。为每个Pod分配固定的网络标识和存储卷，确保数据的持久性。严格按照顺序进行部署和扩缩容，有效防止数据不一致问题。

专门用于运行一次性任务的控制器，如数据库迁移或AI模型训练。智能重试机制确保任务最终完成，可设置并行度提高处理效率。支持配置超时和重试策略，灵活应对各种场景。

Kubernetes中的存储

# 持久卷（PV）
持久卷是Kubernetes中的核心存储抽象，它完全独立于Pod的生命周期，确保数据的持久性和可靠性。作为一种集群级别的资源，PV可用灵活地支持包括NFS、Ceph、各类云存储内的多种存储后端。通过配置ReadWriteOnce、ReadOnlyMany等不同的访问模式，能够精确控制存储资源的使用方式，满足不同应用场景的需求。
# 持久卷声明（PVC）
持久卷声明作为存储资源的请求接口，在Pod和存储系统之间构建了一层优雅抽象层。它使开放团队能够专注于定义存储需求，如容量大小、访问模式和期望的存储类型，而无需关心底层存储的具体实现细节。当创建PVC时，Kubernetes会智能地选择或创建满足要求的持久卷，建立稳定的绑定关系。
# 存储类（StorageClass）
存储类是Kubernetes中动态存储配置的基础，它定义了存储服务的质量特征和配置参数。通过设置不同级别的性能指标，如IOPS限制，吞吐量要求等，可用满足从开放测试到高性能生产环境的各类需求。存储类的引入使集群管理员能够standardize存储配置，提供可预测且一致的存储服务体验。

Kubernetes中的配置管理

# ConfigMap

# Secret
专门设计用于安全存储敏感信息，采用base64编码并支持加密存储。适用于管理数据库用户和密码、API密钥、OAuth认证令牌、TLS/SSL证书等重要凭据。Secret在etcd中以加密形式保存，确保了数据在传输和存储过程的安全性，并支持细粒度的访问控制，防止铭感信息泄露。

# 自定义资源
通过CustomResourceDefinition(CRD)机制扩展Kubernetes原生API能力。使开发团队能够定义符合特定业务场景的资源类型和控制器逻辑。常见应用包括：自动化数据库实例的创建和管理、配置消息队列集群参数、管理自定义监控规则等。CRD极大地提升了Kubernetes的可扩展性，使其能够更好适应复杂的企业级应用场景。

5.1 kubernetes级别概念讲解

集群设计

# 集群规模
生产环境推荐配置5-7个节点构建高可用集群。每个节点建议配置16核CPU、64GB内存核500GB SSD存储。设置节点亲和性规则将负载分散，并配置Cluster Autoscaler在资源利用率超过75%时自动扩容。
# 节点类型
部署3个Master节点（4核16GB），采用etcd集群实现控制平面高可用，使用keepalived+haproxy实现API Server负载均衡。根据应用特点划分Worker节点组：计算密集型（16核64GB）、内存密集型（8核128GB）和通用型（8核32G）。
# 网络架构
采用Calico BGP模式，配置IPIP隧道支持跨网段通信。Pod网络设置为172.16.0.0/16，Service网段为10.96.0.0/12。启用Network Policy实现微服务间访问控制，配置MTU为1440确保网络性能最优。
# 存储方案
核心数据库使用本地NVMe SSD配合LocalPV实现低延迟。通用存储采用Rook管理的Ceph集群，提供RBD和CephFS两种存储类型。配置daily快照策略，通过Velero实现集群级别定时备份，保留期设置为30天。

资源管理

# 资源配额
为每个命名空间配置ResourceQuota: 限制CPU使用不超过16核，内存上限32GB，最大Pod数量50个。为开发环境配置50%配额，生产环境80%配额，预留20%用于系统开销。
# 资源请求
标准Web应用Pod推荐配置：CPU请求0.5核，内存请求512MB；
数据处理Pod配置：CPU请求2核，内存请求4GB。确保请求值为实际使用量的1.5倍，避免资源争抢。
# 资源监控
使用Prometheus采集15秒间隔的资源指标，Grafana设置可视化面板。CPU使用率超过80%持续5分钟、内存使用率超过85持续3分钟时触发告警，并通过HPA自动扩容。配置Loki收集容器日志，保留期30天。
# 资源限制
设置limits/requests比率不超过1.5，如Web应用CPU限制0.75核，内存限制768MB。对数据库等核心服务强制设置limits，CPU最高4核，内存最高8GB，防止系统不稳定。

高可用性

# 多Master节点
部署3-5个Master节点并配置HAProxy或Nginx负载均衡器，使用etcd集群存储配置数据。通过keeplived实现VIP漂移，确保即使损失任意两个Master节点集群仍能正常运行。
# Pod复制
配置Deployment控制器设置replicas=3-5，结合Pod反亲和性（podAntiAffinity）将副本分散到不同节点核可用区。使用PodDisruptionBudget确保更新时至少保持75%的副本可用。
# 故障恢复
配置livenessProbe（30秒间隔）和readinessProbe（10秒间隔）进行健康检查，设置preStop钩子实现优雅关闭。运行Pod在节点故障时自动迁移，配置tolerations参数应对节点维护和网络分区。
# 自动伸缩
启用HPA，设置CPU使用率阈值为70%，内存阈值为80%。配置自定义指标如请求延迟（<500ms）和队列长度监控。根据实际负载在2-10个副本之间自动调节，冷却实践设置为3分钟。

安全配置

# RBAC
通过Kubernetes ClusterRole和RoleBinding实现精细化权限控制，为开发团队配置仅限命名空间的访问权限，为运维团队设置集群级管理权限。定期审计权限分配，确保符合最小权限原则。

# 网络策略
使用Calico网络插件实现微分段，限制Pod间通信仅允许必要的5000-9000端口范围。配置默认拒绝策略，仅允许指定命名空间间的通信，并启用网络流量监控和告警。

# 安全扫描
采用AES-256算法加密etcd存储数据，使用Vault进行密钥管理，为所有服务间通信配置mTLS加密。通过KMS提供者加密Secrets，定期轮换加密密钥实现存储和传输全程加密保护。

# 数据加密
部署Trivy扫描器执行每日镜像扫描，设置Critical级别漏洞阻断策略。通过OPA Gatekeeper强制实施容器镜像签名验证，确保仅使用可信源的基础镜像，并维护CVE修复响应时间不超过48小时。

Ubuntu安装环境

# 选择Ubuntu 20.04 LTS 作为操作系统是部署Kubernetes的最佳基础。
此版本要求最低配置为：2核CPU、4GB内存和50GB磁盘空间。安装过程中需要执行apt update和apt upgrade确保系统最新，并通过apt install docker.io containerd kubernetes-cni安装必要组件。Ubuntu提供的软件包源包含了所有必需的依赖，如容器运行时（Docker 20.10+）和网络插件。得益于其强大的apt包管理系统和全球技术社区支持，您可用使用ubuntu-bug工具快速报告问题，并获得每月的安全补丁更新。特别推荐启用unattended-upgrades来自动应用关键安全更新，确保系统持续稳定和安全。

安装Kubeadm

Kubeadm是Kubernetes的官方集群安装工具，要求系统已安装Docker 20.10.x 或更高版本。首先需要通过apt包管理器安装kubeadm（最新稳定版本1.27.x），运行命令：apt-get update && apt-get install -y kubeadm kubectl kubelet。kubeadm会自动处理关键的部署任务，包括生成集群CA证书、部署核心控制平面组件（如kube-apiserver、etcd、kube-scheduler）、配置Pod网络（支持Calico、Flannel等CNI插件），以及设置RBAC权限策略。这种自动化的方式不仅确保了配置符合Kubernetes安全标准，还将通过需要数小时的复杂部署过程简化为几个简单的命令。

初始化Master节点

在Master节点上执行kubeadm init --pod-network-cidr=10.244.0.0/16 --kubernetes-version=v1.24.0 命令来初始化集群控制平面。此步骤会自动配置核心组件：kube-apiserver负责API调度，以及作为分布式数据库的etcd。系统会生成多个关键文件：admin.conf用于集群管理，controller-manager.conf用于控制器配置，scheduler.conf用于调度器设置，以及包含集群加入令牌的bootstrap-token。这些文件都存储在/etc/kubernetes目录下，确保了集群的安全性和可靠性。

加入Worker节点

在每个Worker节点上执行kubeadm join 命令（例如: kubeadm join 192.168.1.10:6443 --token xxxx.xxxxxxxxxxxx --discovery-token-ca-cert-hash sha256:xxxxxxx），将新节点添加到现有的Kubernetes集群中。这个过程需要使用在初始化Master节点时生成的24小时有效期安全令牌和证书哈希值，确保集群通信的安全性。加入过程包括：安全必要的容器运行时（如Docker或containerd）、配置kubelet服务、建立TLS加密通道，并注册节点到API服务器。成功加入后，可用通过kubectl get nodes命令验证节点状态，确认其是否已准备就绪（Ready）并能接收工作负载。

配置网络

配置容器网络接口（CNI）插件是确保集群正常运行的关键步骤。常用的网络插件包括Calico（适用于大规模部署，支持网络策略）和Flannel（简单易用，适用小型集群）。安装完成后，这些插件会自动为每个Pod分配10.244.0.0/16或192.168.0.0/16等范围内的唯一IP地址。

网络插件的主要功能包括：创建虚拟网络接口、配置路由表、实现跨节点的overlay网络，以及处理IPAM（IP地址管理）。通过这些机制，可以确保不同节点上的Pod能够通过内部DNS服务相互发现，并通过kube-proxy实现服务负载均衡。集群内的所有网络流量都经过加密，支持标准的网络策略进行访问控制。

安装组件

# 按照以下顺序安装Kubernetes v1.24或更高版本的核心组件，确保集群的基础功能正常运行：
1. kubelet: 在每个节点上安装10.x版本的kubelet服务，配置systemd自启动，设置默认内存限制为2GB。它负责与容器运行时（如Docker或containerd）交互，确保容器的健康状态监控和资源管理。
2. kube-apiserver：部署在主节点上，配置安全端口6443和非安全端口8080，启用RBAC授权模式。作为集群的统一入口，它处理所有REST API请求，支持每秒处理多达5000个API调用。
3. kube-controller-manager：配置leader选举机制，设置节点监控周期为5秒。它运行多个控制器进程，包括节点控制器、副本控制器和服务账号控制器，确保集群始终维持在期望状态。
4. kube-scheduler: 启用高级调度策略，配置资源权重比为CPU:Memory=2:1。它通过预选和优选两个阶段，分析节点资源使用情况、亲和性规则和污点容忍度，为新Pod选择最优节点。

验证集群

# 按照以下详细步骤验证集群的运行状态：
1. 运行kubectl get nodes检查所有节点状态，确保节点显示为 "Ready" 状态

4. 测试网络连通性：使用kubectl run nginx --image=nginx部署测试容器，验证Pod是否成功创建和运行。
如果以上所有检查都通过，说明您的Kubernetes集群已经准备就绪，可以开始部署应用程序了。

常见问题与解决方法

# Pod启动失败
常见错误码CrashLoopBackOff或ImagePullBackOff时，首选执行kubectl describe pod [pod-name] -n [namespace] 检查详细状态。对于镜像拉取失败，验证imagePullSecrets配置是否正确，确认私有仓库的访问凭证。如遇内存不足，检查limits设置，建议设置内存请求为限制值的50%-80%。使用kubectl logs [pod-name] -n [namespace] --previous 查看崩溃前的日志记录。

# 网络连接异常
当出现Pod间通信失败时，首先执行kubectl get networkpolicies -n [namespace] 检查访问策略。使用kubectl exec -it [pod-name] -n [namespace] --ping [service-name] 测试网络连通性。对于Calico网络插件，运行calicoctl node status 验证BGP连接状态。检查kube-proxy日志确认访问发现是否正常，可通过kubectl get endpoints [service-name] -n [namespace]验证后端Pod IP映射。

# 资源不足
使用kubectl top nodes --sort-by=cpu.usage 识别高负载节点。当CPU使用率超过85%或内存使用超过80%时及时扩容。设置HPA规则：kubectl autoscale deployment [name] --cpu-percent=70 --min=3 --max=10。对于突发流量，建议预留20%缓冲资源。监控kubelet的驱逐阈值，默认设置 --eviction-hard=memory.available<100Mi,nodefs.available<10%。

# 应用部署失败
使用kubectl rollout status deployment [name] -n [namespace]实时监控部署状态。遇到ProgressDeadlineExceeded错误时，检查readinessProbe配置是否合理，建议设置初始延迟3-5秒。执行金丝雀发布：kubectl patch deployment [name] -p '{"spec": {"strategy": {"rollingUpdate": {"naxSurge":1, "maxUnavailable":0}}}}' 确保每次只更新一个Pod。使用kubectl rollout history deployment [name] --revision=2 对比配置变更，必要时快速回滚。
